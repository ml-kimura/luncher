# Personas

## Persona 1: 当日参加者（一般社員）
- **目的**: 手間なく参加表明し、当日のランチ候補を把握して参加する
- **主な行動**:
  - Slackから参加表明
  - 通知リンクからWebで店舗候補確認
  - ランチ後に簡単なフィードバック投稿
- **課題**:
  - 参加操作が煩雑だと離脱する
  - 店舗情報が不足すると意思決定が遅れる
- **成功条件**:
  - 1分以内に参加登録完了
  - 候補比較情報（人数可否、雰囲気、アクセス）が明確

## Persona 2: ランチ申請者（支払実行者）
- **目的**: レシート処理と経費入力を正確かつ短時間で完了する
- **主な行動**:
  - ランチ後にコラボランチ申請を開始
  - レシート画像アップロード
  - OCR結果確認・補正
  - SaaS入力補助を使って精算登録
- **課題**:
  - OCR誤認識時の修正負荷
  - 補助上限超過の判断ミス
- **成功条件**:
  - OCR後の修正を最小化
  - 上限チェックを自動判定し、差戻しを減らす

## Persona 3: システム管理者（制約・設定管理）
- **目的**: 制度変更をコード修正なしで反映する
- **主な行動**:
  - `system_configs` の有効日付き設定更新
  - マッチング制約（例: 人数、部署制約）の調整
  - 申請制約（例: 1名あたり月2回上限）の調整
  - 運用指標の監視
- **課題**:
  - 設定変更の影響範囲が見えにくい
  - 反映タイミングミスのリスク
- **成功条件**:
  - 変更履歴と有効期間が明確
  - 変更後の挙動が仕様上検証可能
