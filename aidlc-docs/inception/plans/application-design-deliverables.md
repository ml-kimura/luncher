# Application Design フェーズ成果物目次案

## 1. 目的

`docs/specs/docs/ja/1.0.0` の仕様を、設計・実装接続に耐える **情報アーキテクチャ** として固定する。本書は Application Design で整備する成果物の **目次** と **相互参照ルール** を示す。

## 2. 正本とユーザーストーリー範囲

- **ユーザーストーリー（正本）**: `docs/specs/docs/ja/1.0.0/user-stories/us-001.md` 〜 **`us-007.md`（計7本）**
  - `aidlc-docs/inception/user-stories/stories.md` はエピック別の別表現あり。設計マッピングは **ファイル名 `us-00N.md` を正** とする。
- **要件（前提）**: `aidlc-docs/inception/requirements/requirements.md`

## 4. 画面仕様テンプレート（ベースライン）

- **必須セクション**: `requirements.md` FR-03 の9項目（見出しは空でも維持可）。
- **参照例**: `docs/specs/docs/ja/1.0.0/screen/design/login.md`・`home.md`。新規画面も同じ規約（ルート名のファイル名、項目一覧・イベント等は `.cursor/skills/screen-design-spec-writer/SKILL.md`）で書く。

## 5. 相互参照ルール

1. **ユーザーストーリー → 画面/API/バッチ**
   - 各 `us-00N.md` の `outputs`（screen / api / batch / db）に対し、対応する設計ファイルへ **ファイルパスまたは ID でリンク** する。
2. **画面 ↔ API**
   - 画面の「イベント/API連携」に Route または operationId を記載。`docs/specs/docs/public/openapi/openapi-app.yml`（生成物の正）と **名前の双方向整合** を取る。
3. **バッチ ↔ DB**
   - バッチ設計に読み書きテーブルを列挙し、CDM/PDM のエンティティ名と一致させる。
4. **対象外の再掲**
   - 経費の **申請** は実装しない（`requirements.md` §4、`workflow-planning.md` §3 と同趣旨）。該当しうる画面/API は先頭で明記する。
