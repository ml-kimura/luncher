# Units Generation 候補一覧（優先度・依存関係）

## 1. 目的と正本

Construction の実装の束ね方の **候補** である。

- **ストーリー ID の正本**: `docs/specs/docs/ja/1.0.0/user-stories/` の **`us-001.md`〜`us-007.md`（7本）**
- `aidlc-docs/inception/user-stories/stories.md` の INVEST 表は参考。番号や件数が ja 仕様と食い違う場合は **常に ja の `us-00N.md` を優先**する。

## 2. 識別子の考え方（別採番しない）

- **実装のまとまりの主キーは、すでに仕様側にある `us-00N` で足りる**。**`U-001` のような第 2 の番号体系は置かない**。
- コード・OpenAPI・画面・DB などは、**それぞれの成果物が持つ ID**（`operationId`、画面 frontmatter の `id`、テーブル名、パッケージ名など）で追跡する。
- 特定の `us-00N` に紐づかない**横断作業**（DB 土台、Slack アプリ接続、認証の共通部など）は、行ラベル **「基盤（横断）」** として扱い、**新しい通し番号は振らない**。

## 3. ストーリー一覧（要約タイトル）

| ストーリー | タイトル（`us-00N.md` frontmatter 準拠） |
|------------|------------------------------------------|
| `us-001` | 日次のコラボランチ案内を Slack に配信できる |
| `us-002` | Slack から参加表明できる |

| ストーリー | 備考（再分割・境界の確認ポイント） |
|------------|--------------------------------------|
| `us-002` | Slack 対話モデル、締切・**月次上限**判定の切り出し |

## 5. 実装の束ね方（候補）

| 主トラッキング | 名称 | 主な成果物の方向 | 依存（前提） |
|----------------|------|------------------|--------------|
| **基盤（横断）** | 基盤 | DB 初期、Slack クライアント、Web/API 土台、認証・共通エラー | なし（他の前提） |
| `us-001` | 日次募集 | バッチ、Slack 投稿、当日ランチレコード、ログ | 基盤 |
| `us-002` | 参加表明 | Slack インタラクション、参加 API、締切・月次上限 | 基盤、（当日枠なら `us-001` が論理先行） |
| `us-003` | 自動マッチング | バッチ/ジョブ、ドメイン、設定参照 | 基盤、`us-002`、（設定可変なら `us-007` かコードデフォルト） |
| `us-004` | 割当後の欠席 | Slack + API、参加状態、通知 | 基盤、`us-003` |
| `us-005` | グループへ追参加 | Slack + API、定員・通知、参加状態 | 基盤、`us-003` |
| `us-006` | 精算用テキスト（写真リプライ） | Slack イベント、テキスト生成（OCR 等は方針次第） | 基盤、（ランチと紐づければ `us-003` 以降） |
| `us-007` | 管理者の要件設定 | 管理画面、設定 API・永続化 | 基盤 |

**メモ**

- 利用者向け **ログイン・ホーム** は基盤＋各ストーリーに内包可能。管理者 UI は **`us-007`**。
- **経費の申請**は扱わない（`workflow-planning.md` §3・`requirements.md` §4）。`us-006` は**申請補助**の範囲。

## 6. MVP 順（提案）

「募集 → 参加 → 編成」中心の**一例**（ラベルは上表と同じ）：

1. 基盤（横断）  
2. `us-001`  
3. `us-002`  
4. `us-003` — 設定はデフォルトでも可。**設定駆動の検証を最初からするなら `us-007` を 4 の前 or 4 と並行**  
5. `us-004` / `us-005`  
6. `us-006`  
7. `us-007`（上の 4 で未手当なら前倒し）  

新しい画面や機能は、**新規の `us-00N` を `user-stories` に追加してから**同じ表の行として足す（**別の通し ID は増やさない**）。

## 7. 依存関係（テキスト）

- `us-003` は `us-002` のデータに依存。  
- `us-004` / `us-005` は `us-003`（グループ状態）に依存。`us-005` は `us-004` とも連携しうる。  
- `us-006` は基盤だけで始め可能だが、ランチと紐づければ `us-003` 以降と仕様上つなげる。  
- `us-007` は `us-003` の制約入力の**保存先**になりうる。デフォルトで `us-003` 先行の場合は、後追いで `us-007` でも可。

## 8. Construction への引き継ぎ

- 作業のトラッキングは **Issue / ブランチ / ディレクトリ** など、**リポジトリの既存の付け方**に合わせる。  
- 本書は **`us-00N` 単位**で何を先に作るかの**順序と依存**のメモに留まる。`aidlc-docs/construction/…` を使う場合も、**フォルダ名を `us-001` などストーリー ID に合わせる**、など、**二重採番を作らない**。

依存やストーリーが変わった場合、本ファイルと `application-design-deliverables.md` を同期する。

**最終メンテ**: `docs/specs/docs/ja/1.0.0/user-stories/us-001`〜`us-007` および `requirements.md`（§3・§4・FR-06 等）に整合させる。
