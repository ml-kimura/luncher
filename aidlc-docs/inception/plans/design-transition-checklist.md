# 設計移行前チェックリスト（Quality Gate）

## 1. 適用タイミング

Application Design のレビュー完了時、および Construction に入る直前に実施する。  
拡張 **Security Baseline** / **Property-Based Testing** が `aidlc-state.md` で有効な場合、§4〜§5 は **適用必須**（該当なしは「N/A」と理由を記載）。

## 2. 仕様整合性

| # | 項目 | 確認内容 |
|---|------|----------|
| S1 | ストーリー網羅 | `us-001.md`〜`us-007.md` の**受け入れ条件**（Acceptance Criteria。frontmatter の `acceptanceCriteria` 等）が、対応する画面/API/バッチのいずれかでトレース可能 |
| S2 | 対象外の明示 | 経費の「申請」機能が仕様に混入していない（「申請補助」は対象内と混同していない） |
| S3 | 制約の反映 | 月次上限・締切・管理者変更が保存されるべき場所がドキュメント上で一貫 |
| S4 | リンク | 仕様間リンクが切れていない（相対パス、ファイル名規則） |
| S5 | 用語 | 画面/API/要件で同一概念の名称が揃っている |
| S6 | 例外系 | 主要ユースケースに対応する異常系・権限エラーが記載されている |

## 3. 画面・API・バッチ

| # | 項目 | 確認内容 |
|---|------|----------|
| U1 | 画面テンプレート | `requirements.md` FR-03 のセクションが各画面ファイルで維持されている |
| U2 | OpenAPI との対応 | API 補足と実装側デコレータ/OpenAPI の operation が名前・パラメータで対応 |
| U3 | バッチの境界 | スケジュール、冪等性、失敗時リトライ/アラートが記載されている |

## 4. Security Baseline（拡張が有効な場合）

| # | 項目 | 確認内容 |
|---|------|----------|
| SEC1 | 認証・認可 | Slack リクエストの署名検証、管理画面と API の権限モデルが文書化されている |
| SEC2 | シークレット | トークン類がリポジトリに載らない運用であり、保管場所が決まっている |
| SEC3 | 入力検証 | 外部入力（Slack、ファイルアップロード、フォーム）に検証方針がある |
| SEC4 | PII | US-006 等で個人情報・画像の保持・削除が要件レベルで触れられている |

## 5. Property-Based Testing（拡張が有効な場合）

| # | 項目 | 確認内容 |
|---|------|----------|
| PBT1 | 対象領域の明示 | マッチング制約・数値パース・設定バリデーションなど、プロパティ候補が設計書または単位計画に記載されている |
| PBT2 | 境界値 | 最小人数・上限値・空リストなど、プロパティで押さえる境界が要件から読み取れる |

## 6. フェーズ移行基準

次をすべて満たすとき **Construction（または Units Generation の確定）へ進める**。

1. §2 の項目がすべて **チェック済みまたは欠落理由が記録されている**。
2. §3 でブロッカー（テンプレ欠落、API と画面の矛盾）がない。
3. §4〜§5 は拡張が有効なら対応済み、無効または該当なしなら **N/A と理由** が残っている。

レビュー記録は PR コメントまたは `aidlc-docs` 内のレビューログに残す運用としてよい。
