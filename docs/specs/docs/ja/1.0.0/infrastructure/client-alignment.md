# クライアント事前調整

アプリ専用テナントに **ゲストとして招待する**方式では、基本的に本アプリ側テナントで認証・認可を管理できる。クライアント企業側ポリシーにより、次の項目は事前調整が必要になる場合がある。出典: `infra/blueprint.md` の「要調整事項」。

| 要調整項目                             | 発生しやすいケース                                                      | 利用者に見える症状                                          | クライアント企業側への依頼内容（具体）                                                                | 事前確認の方法                                     |
| -------------------------------------- | ----------------------------------------------------------------------- | ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| **外部コラボレーション許可**           | 外部組織との B2B 招待受け入れをデフォルト拒否している                   | 招待の redeem に失敗、`組織ポリシーで拒否` 相当             | 本アプリ用テナント（tenant ID / domain）を外部コラボ許可対象に追加する                                | テストユーザー１名を事前に招待し redeem 可否を確認 |
| **クロステナントアクセス（Outbound）** | Cross-tenant access の Outbound で対象テナントへの B2B が拒否されている | サインイン直後に失敗、`AADSTS` 系エラー                     | Outbound で本アプリ用テナントへの B2B collaboration を許可する                                        | 本番同等ポリシー下のサインイン試験                 |
| **MFA / 条件付きアクセス**             | 外部 SaaS にも端末準拠や特定認証強度を必須化している                    | `MFA 不足` / `Compliant device required`、Twingate でも同様 | 対象アプリ（本アプリ / Twingate）に適用する CA ポリシーを明示し、必要なら例外または要件緩和を設定する | 「社給端末」「私物端末」それぞれでログイン試験     |
| **外部 MFA 信頼方針**                  | ホームテナント MFA をリソース側で信頼しない設計                         | 二重 MFA または認証強度不一致で失敗                         | クロステナント信頼（外部 MFA を信頼するか）と認証強度要件を相互に合意する                             | MFA１回で通るか、追加チャレンジ有無を確認          |
| **許可ユーザー範囲**                   | 外部利用を特定ユーザー/グループのみに制限している                       | 一部ユーザーのみ利用可                                      | 対象ユーザー（またはグループ）を事前確定し、外部利用許可と招待対象を一致させる                        | 代表３パターン（管理者・一般・対象外）でテスト     |

## クライアント企業への依頼テンプレート（最小）

- 本アプリ用テナントの `tenant ID` と `primary domain` を共有し、B2B 受け入れ許可を依頼する。
- Cross-tenant access の Outbound で、上記テナント宛て B2B collaboration を許可する。
- 本アプリおよび Twingate へのアクセス時に適用する条件付きアクセスポリシー（MFA、端末準拠、認証強度）を確定する。
- 利用対象ユーザー/グループを確定し、対象者で事前接続テスト（招待 redeem、アプリ、Twingate）を実施する。

**運用上の推奨**: オンボーディング時は「テナント許可」「条件付きアクセスポリシー」「対象ユーザー」「接続テスト結果」を１シートにまとめ、双方承認を完了条件にする。Twingate の IdP を本アプリ用テナントに統一する場合も、ホームテナント側ポリシー影響は残る前提で確認する。
