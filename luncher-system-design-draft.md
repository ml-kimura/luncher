# システム設計書：社内コラボランチ・マッチングシステム (Complete Edition)

## 1. 解決する課題 (Problem Statement)

- **組織のサイロ化の解消**: 部署を越えたカジュアルな交流機会を自動創出し、横の繋がりを強化する。
- **ランチ選定の認知的負荷**: 4名以上の予約しやすさ、会話に適した環境など、条件に合う店を探す手間を排除する。
- **経費精算の心理的障壁**: レシート管理や手入力の煩雑さを解消し、制度利用のハードルを下げる。
- **ルールの柔軟な運用**: 補助金額やマッチングロジックの変更を、コード修正なしで迅速に反映させる。

## 2. ユースケース (Use Cases)

- **一般社員**: Slackで手軽に参加表明し、Webアプリで「社内推奨」の店舗を確認。ランチ後はSaaSへの自動入力補助で精算。
- **支払担当者**: Web画面からレシートをアップロードし、OCR結果を確認してSaaSへデータを流し込む。
- **管理者**: `system_configs` テーブルを更新し、来月からの補助額アップやマッチングロジックの調整を予約設定する。

## 3. システムアーキテクチャ (Architecture)

- **Authentication (IdP)**: **Slack OAuth (via Supabase Auth)**
  - Slack User ID をユニークキーとして全テーブルで共通化し、パスワードレスなSSOを実現。
- **Messaging**: **Slack** (エントリーの受付、マッチング結果のプッシュ通知)
- **Frontend**: **Next.js** (店舗登録、詳細閲覧、レシートOCR、SaaS入力補助UI)
- **Backend**: **Supabase** (PostgreSQL, Edge Functions, Storage for receipts)
- **External API**: Google Maps API (店舗検索), AI/OCR API (レシート解析)

## 5. 主要ロジックと機能

### A. 自動マッチングロジック

- **必須制約 (Hard)**:
  - 4名以上の確保、同一部署の重複回避。
  - 1名あたり月２回まで。
- **優先スコア (Soft)**:
  - `new_encounter`: 過去の同席履歴がないペアを優先。
  - `age_diversity`: 異なる年齢層の組み合わせ。
- **パラメータ化**: これらの重み付けは `system_configs` から動的にロード。

### B. ハイブリッド店舗提案

- **Standard (定番)**: 社内評価が高く、過去の利用実績が豊富な「安心できる」店。
- **Discovery (発見)**: 社内利用実績が少ない、またはメンバー全員が未訪問の「新規開拓」店。

### C. 経費精算補助 (SaaS連携)

- **入力補助 (Pre-fill)**: SaaSにログインした状態でWeb画面からデータを流し込む。
- **バリデーション**: `system_configs` の当時の上限額に基づき、補助条件を自動チェック。

## 6. 運用フロー

1. **エントリー**: Slackで参加ボタンをクリック。
2. **マッチング**: 指定時刻に Edge Function が起動。ルールに基づきグループ化し、Slack通知。
3. **Web遷移**: 通知URLからWebへ遷移し、店舗の提案内容を確認。
4. **精算**: ランチ後、レシートをWebへアップロード。SaaSへの入力補助を実行。
5. **フィードバック**: Web上で「コラボランチ適性」に特化した口コミを投稿。
